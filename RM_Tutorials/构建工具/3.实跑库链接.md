# 实践库链接

## 库链接到底在干什么

你写了 `target_link_libraries(run opencv_world)`，然后它工作了。但你有没有想过背后发生了什么？

链接器在做一件很实际的事：把你的目标文件里**用到的符号**（函数名、变量名），去其他目标文件或库里**找到定义**，然后把它们**接起来**。

符号找不到，就 undefined reference。找到重复的，就 multiple definition。听起来简单，但实际操作中坑很多。

---

## 符号解析顺序

链接器处理输入文件的顺序是**从左到右**。

```bash
g++ main.o -lopencv_world -lpthread
```

这里 main.o 排在最前面，然后 opencv_world，然后 pthread。

链接器维护一个"未定义符号表"。初始时它是空的。处理 main.o 时，遇到调用的函数（比如 `cv::imread`），就把它加到未定义符号表。然后处理 opencv_world，这个库提供了 `cv::imread` 的实现，所以把它从未定义表里划掉。继续处理 pthread，如果还有未定义的符号，再去找。

**所以顺序很重要。** 如果 A 库依赖 B 库，A 得写在 B 前面。

很多人遇到过：

```
undefined reference to `cv::imread'
```

然后疯狂装库、改 CMakeLists。但第一步应该是检查：库有没有被正确链接？链接顺序对不对？

---

## 静态库和动态库

Linux 上你遇到的 .so 和 .a 文件，分别是动态库和静态库。

### 静态库（.a）

本质上是多个 .o 文件打包。链接时把代码**复制**进去。

```bash
ar rcs libdetector.a detector1.o detector2.o
g++ main.o libdetector.a -o run
```

### 动态库（.so）

链接时不复制，只记录依赖。运行时加载。

```bash
g++ -fPIC -shared detector1.cpp detector2.cpp -o libdetector.so
g++ main.cpp -L. -ldetector -o run
```

`-fPIC` 是 Position Independent Code，动态库需要，因为加载地址不确定。`-shared` 表示生成共享库。

动态库的优点是多个程序共享一份代码，节省空间，而且可以替换库文件而不用重新编译程序。但缺点是部署时得确保 .so 文件在系统能找到的地方。

## 库搜索路径

### 编译时搜索

`-L` 加的路径，只在**编译时**生效。

```bash
g++ main.cpp -L/home/meko/lib -ldetector -o run
```

但这个路径**不会写入可执行文件**。运行时系统不知道去这里找 .so。

### 运行时搜索

Linux 用 `LD_LIBRARY_PATH` 环境变量找 .so：

```bash
export LD_LIBRARY_PATH=/home/meko/lib:$LD_LIBRARY_PATH
./run
```

或者把 .so 放到系统默认路径：`/usr/lib`、`/usr/local/lib`。

更正规的做法是用 `rpath`，把库的路径直接写入可执行文件：

```bash
g++ main.cpp -L/home/meko/lib -ldetector -Wl,-rpath,/home/meko/lib -o run
```

`-Wl,-rpath,path` 把 rpath 传给链接器。这下运行 ./run 的时候不用设置 LD_LIBRARY_PATH 了。

**CMake 里的等效写法：**

```cmake
target_link_libraries(run
    /home/meko/lib/libdetector.so
)

set_target_properties(run PROPERTIES
    BUILD_RPATH "/home/meko/lib"
)
```

### 检查依赖

想知道一个可执行文件依赖哪些 .so，链接了哪些路径？

```bash
ldd ./run
```

输出类似：

```
linux-vdso.so.1 (0x00007fff...)
libopencv_world.so.4.6 => /usr/local/lib/libopencv_world.so.4.6
libpthread.so.0 => /lib/x86_64-linux-gnu/libpthread.so.0
libstdc++.so.6 => /usr/lib/x86_64-linux-gnu/libstdc++.so.6
```

如果某个 .so 显示 "not found"，那就是运行时找不到。常见于交叉编译后拷贝到 ARM 板子，但板子没装对应的库。

## 头文件 vs 库文件

新手经常搞混这两个。

**头文件（.h/.hpp）** 告诉你"怎么用"——函数声明、类定义、模板。编译时需要。

**库文件（.so/.a）** 给你"实际代码"——函数实现。链接时需要，运行时也可能需要。

有时候装了库，但没装 dev 包。Ubuntu 上经常这样：

```bash
sudo apt install libopencv-dev   # 开发包，含头文件+库
# 不是
sudo apt install libopencv4.2    # 运行时包，只有 .so
```

没装 dev 包，编译时报 `fatal error: opencv2/opencv.hpp: No such file or directory`。
装了 runtime 包，链接时报 `undefined reference to cv::imshow`。

先确认装的是开发包。

## CMake 里的 find_library

如果你有自己写的库，不想装到系统路径，可以用 `find_library`：

```cmake
find_library(DETECTOR_LIB
    NAMES detector
    PATHS /home/meko/lib
)

if(NOT DETECTOR_LIB)
    message(FATAL_ERROR "detector library not found")
endif()

target_link_libraries(run ${DETECTOR_LIB})
```

`find_library` 会找 `libdetector.so` 或 `libdetector.a`，返回完整路径。

## 链接顺序和循环依赖

### 顺序问题

如果 a 依赖 b，b 依赖 c：

```bash
g++ main.o -la -lb -lc    # 对
g++ main.o -lc -lb -la    # 错，c 放在最前面，a 可能找不到 b 的符号
```

### 循环依赖

a 依赖 b，b 也依赖 a？静态库不允许循环，链接器会报 `undefined reference`。

解决方案：
1. 把 a 和 b 合并成一个库
2. 用动态库
3. 多次链接（先 -la -lb，再 -la）

## visibility 设置

编译动态库时，建议设置符号可见性：

```cmake
set_target_properties(detector PROPERTIES
    CXX_VISIBILITY_PRESET hidden
    VISIBILITY_INLINES_HIDDEN ON
)

target_compile_options(detector PRIVATE -fvisibility=hidden)
```

这让所有符号默认隐藏，只有标记了 `__attribute__((visibility("default")))` 的才导出。

好处：
- 避免符号冲突（两个库都定义了同名函数）
- 加快动态库加载
- 减少导出的符号数量，更干净

## RM 视觉项目里的链接问题

RM 代码通常依赖 OpenCV、Boost、可能还有 NVIDIA 的 GPU 库。

### 典型 CMake 配置

```cmake
find_package(OpenCV REQUIRED)
find_package(Boost REQUIRED system thread)

add_executable(run main.cpp detector.cpp)

target_include_directories(run PRIVATE
    ${CMAKE_SOURCE_DIR}/include
    ${OpenCV_INCLUDE_DIRS}
    ${Boost_INCLUDE_DIRS}
)

target_link_libraries(run
    ${OpenCV_LIBS}
    ${Boost_LIBRARIES}
    pthread
    dl
)
```

### 找不到 Boost

Boost 库的命名有时候很怪。比如 `boost_regex` 对应 `libboost_regex.so`。`find_package(Boost)` 会自动处理大部分，但有时候你得手动指定：

```cmake
set(Boost_LIBRARY_DIR /usr/local/lib)
find_package(Boost 1.70 REQUIRED)
```

### 多版本 OpenCV 冲突

机器上可能装了 OpenCV 2 和 OpenCV 4，pkg-config 找到的是其中一个。

```bash
pkg-config --modversion opencv4
pkg-config --cflags opencv4
pkg-config --libs opencv4
```

或者直接指定路径：

```cmake
set(OpenCV_DIR /opt/opencv4/lib/cmake/opencv4)
find_package(OpenCV 4 REQUIRED)
```

## 一个实际调试案例

我见过有人 CMakeLists.txt 写得没问题，但链接就是报 undefined reference。

最后发现是：他用的 OpenCV 是 **无 GUI 版本**，没编译进 `cv::imshow`、`cv::waitKey` 这些函数。

这不是 CMake 配置问题，是 OpenCV 本身就没编译这些模块。解决方案只能是重新编译 OpenCV，加上 `-DWITH_QT=ON` 或 `-DWITH_OPENGL=ON` 这些选项。

**教训：找链接错误原因时，先确认库本身是不是真的提供了那个符号。**

用 `nm -C libopencv_world.so | grep imshow` 可以查符号。

## 小技巧

`--start-group` 和 `--end-group` 可以处理循环依赖的库组：

```bash
g++ main.o -Wl,--start-group -la -lb -lc -Wl,--end-group -o run
```

CMake 里：

```cmake
target_link_libraries(run
    -Wl,--start-group
    -la -lb -lc
    -Wl,--end-group
)
```

这在链接大型项目时有用，但也说明你的库设计可能有问题，不建议长期依赖。

---

链接这件事，说到底就是"把用到的代码找到并接上"。弄清楚头文件和库文件的区别、编译时和运行时的区别，百分之九十的链接问题都能自己解决。
