# 编译 C++ 项目

## 你见过多少种编译方式？

有的项目用 CMake，有的用 Makefile，有的用 qmake，有的直接给你脚本。

RM 视觉项目基本上是 CMake。搞清楚怎么编译 CMake 项目，是真有用的技能。

---

## CMake 项目的标准流程

```bash
git clone https://github.com/MEKO747/RobotVision.git
cd RobotVision
mkdir build && cd build
cmake ..
make -j$(nproc)
```

这套流程你可能见过无数遍了。但每一步在干什么？

### mkdir build && cd build

**不要在源码目录直接 build。**

CMake 会生成一堆 Makefile、cmake_install.cmake、CTestTestfile.cmake 之类的文件。如果这些混在源码里，git 会把它们当作文本跟踪，混乱不堪。

而且如果想 clean，重新生成 CMake 配置，直接删掉整个 build 目录就行。

### cmake ..

`cmake ..` 读取上级目录的 CMakeLists.txt，生成 Makefile 到当前目录（build）。

`.` 代表当前目录，`..` 代表上级目录。

如果你想指定安装路径、编译器版本、构建类型：

```bash
cmake .. \
    -DCMAKE_BUILD_TYPE=Release \
    -DCMAKE_INSTALL_PREFIX=/usr/local \
    -DCMAKE_CXX_COMPILER=g++-11
```

### make -j$(nproc)

读取 Makefile，调用 g++ 编译每个目标。`-j$(nproc)` 让 make 并行编译，用所有 CPU 核心加速。

**构建类型（CMAKE_BUILD_TYPE）** 影响优化级别和调试信息：
- Debug：不优化，加调试符号（`-g -O0`）
- Release：优化，不加调试符号（`-O3`）
- RelWithDebInfo：优化但保留调试符号（`-O2 -g`）
- MinSizeRel：优化体积

## Makefile 基础

即使你用 CMake，了解一点 Makefile 也有用。

一个最简单的 Makefile：

```makefile
run: main.cpp
    g++ main.cpp -o run
```

注意**缩进是 Tab，不是空格**。写错缩进会报 `missing separator`。

执行 `make run`，如果 main.cpp 比 run 新，才会重新编译。

```makefile
OBJ = main.o detector.o tracker.o

run: $(OBJ)
    g++ $(OBJ) -o run

main.o: main.cpp detector.h
    g++ -c main.cpp -o main.o

detector.o: detector.cpp detector.h
    g++ -c detector.cpp -o detector.o

tracker.o: tracker.cpp tracker.h
    g++ -c tracker.cpp -o tracker.o

clean:
    rm -f $(OBJ) run
```

Makefile 的核心是**规则**：

```
目标: 依赖
    命令
```

当依赖比目标新，才执行命令。

`-j` 选项同样适用于 make：

```bash
make -j4    # 4 个并行 job
```

## 交叉编译：给 ARM 板子编译

RM 的视觉模组可能是 ARM64 架构。你的开发机是 x86_64。这是交叉编译。

需要 aarch64-linux-gnu-g++ 编译器。

```bash
cmake .. \
    -DCMAKE_C_COMPILER=aarch64-linux-gnu-gcc \
    -DCMAKE_CXX_COMPILER=aarch64-linux-gnu-g++ \
    -DCMAKE_SYSROOT=/path/to/arm/sysroot
```

还需要告诉 CMake 用 cross compile 模式，不跑原生的测试：

```cmake
set(CMAKE_SYSTEM_NAME Linux)
set(CMAKE_CROSSCOMPILING TRUE)
```

CMakeLists.txt 里通常已经有判断逻辑处理这件事。

**常见坑：sysroot 设置不对，链接时找不到 glibc、libstdc++。**

需要把 ARM 版的 C++ 标准库和 glibc 拷贝到 sysroot 里，或者用 sysroot 指向已有的 ARM 根文件系统。

## conan / vcpkg：依赖管理

手动下载、编译、安装第三方库是噩梦。Conan 和 vcpkg 是两个流行的 C++ 包管理器。

### Conan 示例

```bash
conan install .. \
    -if build \
    -s build_type=Release \
    -s compiler.libcxx=libstdc++11 \
    -pr ./profile/armv8
```

这会下载 OpenCV、Boost 等库，安装到 `.conan` 目录，然后 CMake 自动找到它们。

### vcpkg 示例

```bash
./vcpkg install opencv4:x64-linux
cmake -B build -S . -DCMAKE_TOOLCHAIN_FILE=/path/to/vcpkg/scripts/buildsystems/vcpkg.cmake
```

vcpkg 会把库安装到 `vcpkg_installed` 目录，CMake 通过 toolchain file 自动配置。

**两者选哪个？**

我见过更多 RM 项目用 vcpkg，配置简单。但 Conan 的 profile 机制对交叉编译支持更好。看你熟悉哪个。

## 安装和打包

编译完了，想把项目安装到系统路径：

```bash
sudo make install
```

这会：
- 把可执行文件拷贝到 `/usr/local/bin`
- 把头文件拷贝到 `/usr/local/include`
- 把库文件拷贝到 `/usr/local/lib`

如果想安装到指定目录而不是系统路径：

```bash
cmake .. -DCMAKE_INSTALL_PREFIX=/home/meko/robotics
make install
```

## 测试：CTest

CMake 内置了 CTest 测试框架。

```cmake
enable_testing()

add_test(NAME MyTest COMMAND ./run --test)
```

编译后运行：

```bash
ctest
```

或者只跑某个测试：

```bash
ctest -R MyTest
```

有些 RM 项目没写单元测试，这很正常。但如果写代码，建议加上基本的测试，至少编译后自动跑一遍看有没有 crash。

## 分步调试编译问题

### 1. CMake 阶段报错

先看 CMake 输出。它会告诉你：
- 找不到某个库
- 编译器版本不满足
- 某些选项冲突

加上 `-DCMAKE_VERBOSE_MAKEFILE=ON` 让 CMake 显示详细配置过程：

```bash
cmake -DCMAKE_VERBOSE_MAKEFILE=ON ..
```

### 2. 编译阶段报错

编译阶段是 g++ 在跑。看报错是：
- syntax error：代码写错了
- undefined reference：链接时找不到符号
- cannot find -lxxx：库路径没配对

Undefined reference 的解决办法：
1. 确认库有没有正确 `find_package` 或 `find_library`
2. 确认 `target_link_libraries` 有没有加这个库
3. 确认链接顺序（依赖库放后面）

### 3. 链接阶段报错

如果是动态库找不到：

```bash
ldd ./run
```

看哪些 .so 显示 "not found"。

如果库在非标准路径：
1. 加 `LD_LIBRARY_PATH`
2. 或用 `rpath` 重新链接
3. 或把 .so 拷贝到 `/usr/local/lib` 再 `sudo ldconfig`

### 4. 运行时报错

Segmentation fault、assertion failed...

用 gdb：

```bash
gdb ./run
(gdb) run
# crash
(gdb) bt     # backtrace，看调用栈
```

CMake 要是没加调试符号（Release 模式），gdb 里只能看到寄存器，看不到变量名。

## 典型 RM 视觉项目的 CMake 结构

RM 视觉项目通常有参考装甲板识别、摄像头读取、串口通信等模块。

一个简化结构：

```
RobotVision/
├── CMakeLists.txt
├── package.xml              # ROS 包配置
├── include/
│   ├── detector.h
│   └── ...
├── src/
│   ├── main.cpp
│   ├── detector.cpp
│   └── ...
├── config/
│   └── param.yaml
└── scripts/
    └── ...
```

顶层 CMakeLists.txt：

```cmake
cmake_minimum_required(VERSION 3.10)
project(RobotVision)

set(CMAKE_CXX_STANDARD 17)
set(CMAKE_CXX_STANDARD_REQUIRED ON)

find_package(OpenCV 4 REQUIRED)
find_package(Boost 1.70 REQUIRED system thread)
find_package(Eigen3 REQUIRED)

include_directories(
    ${CMAKE_SOURCE_DIR}/include
    ${OpenCV_INCLUDE_DIRS}
    ${Boost_INCLUDE_DIRS}
    ${EIGEN3_INCLUDE_DIRS}
)

add_subdirectory(src/detector)
add_subdirectory(src/camera)
add_subdirectory(src/serial)

add_executable(run
    src/main.cpp
)

target_link_libraries(run
    detector
    camera
    serial
    ${OpenCV_LIBS}
    ${Boost_LIBRARIES}
)
```

## 编译速度优化

项目大了，全量编译很慢。几个提速办法：

### 1. ccache

编译缓存，重复编译时复用之前的结果：

```bash
sudo apt install ccache
cmake .. -DCMAKE_CXX_COMPILER=ccache
make -j$(nproc)
```

第二次编译同一个目标，速度快很多。

### 2. 预编译头

如果你有一些基本不变的头文件（Eigen、OpenCV core），可以预编译：

```cmake
set_target_properties(run PROPERTIES
    COMPILE_FLAGS "-H"
)
```

但 Eigen 这种模板库，预编译收益不大，而且模板实例化差异可能导致问题。

### 3. Unity Build

把多个 .cpp 文件合并成一个编译，减少重复解析头文件的开销：

```cmake
set(CMAKE_UNITY_BUILD ON)
set(CMAKE_UNITY_BUILD_BATCH_SIZE 16)
```

CLion 用这个比较多。

## 部署到 RM 机器

交叉编译完成后，拷贝到板子上：

```bash
scp -r build/run meko@192.168.1.120:/home/robot/
```

注意：
1. 确认是静态链接还是动态链接。动态链接的话板子得有对应版本的 .so。
2. 确认架构匹配（aarch64）。
3. 如果用了硬浮点 ABI，确认板子支持。

有时候本地编译完全正常，拷贝到板子跑不起来。常见原因：
- 动态库找不到
- 硬浮点/软浮点不一致
- glibc 版本板子太低

## 写在最后

编译这件事，说到底是把人类能看懂的代码，变成机器能执行的指令。

从手动调用 g++，到写 Makefile，到用 CMake 管理大项目，这条路是必经的。

别怕报错。编译报错是最诚实的报错——代码有毛病，机器不会替你瞒着。

看报错信息，一步一步定位。CMake 报错看 CMake 输出，编译报错看 g++ 输出，链接报错看 ld 输出，运行时报错用 gdb。

耐心一点，问题总能解决。
