# 学习 CMake 基础

## 你第一次打开别人的 C++ 项目时什么感觉？

反正我第一次看到 CMakeLists.txt 是懵的。

一堆 `cmake_minimum_required`、`project`、`add_executable`，
然后 `target_link_libraries`，然后没了。

但这到底是怎么变成可执行文件的？

---

## CMake 是什么

CMake 不是一个编译器。它是一个**构建系统生成器**。

你写一个 `CMakeLists.txt`，CMake 读取它，然后**生成**你平台原生的构建文件——在 Linux 上生成 Makefile，在 Windows 上可能生成 VS 项目。

为什么这么做？因为不同平台、不同IDE的"项目文件"格式完全不一样。你不想为每个平台单独写一套配置。

CMake 就是这个中间层。你写一套配置，全平台通用。

---

## 最小 CMake 项目

你只需要两个东西：一个 C++ 源文件，一个 CMakeLists.txt。

```cpp
// main.cpp
#include <iostream>

int main() {
    std::cout << "Hello, RoboMaster!" << std::endl;
    return 0;
}
```

```cmake
# CMakeLists.txt
cmake_minimum_required(VERSION 3.10)
project(RobotVision)

add_executable(run main.cpp)
```

就这些。保存，然后：

```bash
mkdir build
cd build
cmake ..
make
./run
```

你可能好奇为什么要建 `build` 目录。这是因为 CMake 生成的构建文件会放在这里，**不要在源代码目录直接构建**。很多人踩过这个坑——build 目录下全是生成的临时文件，污染源码。

---

## target 这个概念很关键

你可能注意到 `add_executable` 前面加了 `target_`。

CMake 3.x 之后的写法都是 `target_xxx`，因为它代表一个**目标**。

目标可以是：
- 可执行文件（`add_executable`）
- 静态库（`add_library` 配合 `STATIC`）
- 动态库（`add_library` 配合 `SHARED`）

为什么要用 target？因为 target 可以附加各种**属性**和**依赖**。

```cmake
add_executable(run main.cpp)

target_include_directories(run PUBLIC ${CMAKE_SOURCE_DIR}/include)
target_link_libraries(run PUBLIC opencv_world)
```

这样 `run` 这个目标就知道了：
- 头文件在哪（`include_directories`）
- 链接什么库（`link_libraries`）

PUBLIC/PRIVATE/INTERFACE 这三个修饰符很多人搞晕。说白了就是：
- **PUBLIC**：自己和依赖我的人都要用
- **PRIVATE**：只有自己用
- **INTERFACE**：只有依赖我的人用

一个常见错误是给静态库加了 PRIVATE 的 include directories，然后可执行文件链接这个库时报头文件找不到。

## find_package 才是真正的力量

你不太可能从零写所有代码。RM 视觉代码肯定要用 OpenCV、Boost 这些库。

CMake 的 `find_package` 就是帮你找到系统里已安装的库：

```cmake
cmake_minimum_required(VERSION 3.10)
project(RobotVision)

# 找 OpenCV
find_package(OpenCV REQUIRED)

# 找 Eigen（纯头文件的线性代数库）
find_package(Eigen3 REQUIRED)

add_executable(run main.cpp detector.cpp)

target_include_directories(run PUBLIC
    ${CMAKE_SOURCE_DIR}/include
    ${OpenCV_INCLUDE_DIRS}
    ${EIGEN3_INCLUDE_DIRS}
)

target_link_libraries(run ${OpenCV_LIBS})
```

`find_package` 会设置一堆变量如 `OpenCV_INCLUDE_DIRS`、`OpenCV_LIBS`。不同库的变量名不同，得查文档。

很多新手问：为什么我 include 一直报错？先检查 `find_package` 有没有成功。CMake 会输出类似：

```
-- Found OpenCV: /usr/local/lib
```

如果没找到，会报 fatal error。你可能得手动指定路径：

```bash
cmake -DOpenCV_DIR=/path/to/opencv/build ..
```

## 变量和流程控制

CMake 自己的语法是脚本式的。

```cmake
set(PROJECT_NAME "RobotVision")
set(SOURCES main.cpp detector.cpp tracker.cpp)

message(STATUS "Building ${PROJECT_NAME}")
message(STATUS "Sources: ${SOURCES}")
```

流程控制：

```cmake
if(WIN32)
    set(PLATFORM_LIBS winmm.lib)
elseif(UNIX)
    set(PLATFORM_LIBS pthread)
endif()
```

你可能觉得这像 shell 脚本。确实有点。但记住 CMake 是在**配置阶段**执行的，不是在编译阶段。

---

## 生成器表达式

这是进阶内容，但 RM 视觉项目里经常见到：

```cmake
target_compile_options(run PRIVATE
    $<$<CONFIG:Debug>:-O0 -g>
    $<$<CONFIG:Release>:-O3>
)
```

`$<...>` 这种东西叫生成器表达式，它在**生成构建系统**时才展开。Debug 配置就加 `-O0 -g`，Release 配置就加 `-O3`。

## 常见坑

**1. 头文件不编译进去**

```cmake
add_executable(run main.cpp)
```

只写了 `main.cpp`，但 `detector.h` 不会被自动包含。CMake 不会分析你的 `#include`。你得：

```cmake
target_sources(run PRIVATE detector.h)
```

或者直接：

```cmake
add_executable(run
    main.cpp
    detector.h
    detector.cpp
)
```

**2. 库链接顺序**

链接器是从左到右处理的，左边的库依赖右边的库。如果你报 undefined reference，先检查链接顺序。

**3. TARGET 拼写错误没人报**

```cmake
target_include_directorues(run PUBLIC ...)  # 拼错了 directorue
```

CMake 不会报错，它会创建一个新的、不存在的 target。最终编译时报一堆 undefined reference 才反应过来。

## RM 视觉项目里的 CMake

RM 视觉项目通常不小。几个摄像头、OpenCV、算法模块分开。

典型结构：

```
RobotVision/
├── CMakeLists.txt
├── src/
│   ├── main.cpp
│   ├── detector/
│   │   ├── CMakeLists.txt
│   │   ├── detector.cpp
│   │   └── detector.h
│   └── tracker/
│       ├── CMakeLists.txt
│       └── tracker.cpp
├── include/
│   └── ...
└── build/
```

每个子模块有自己的 CMakeLists.txt，顶层用 `add_subdirectory` 包含：

```cmake
# 顶层 CMakeLists.txt
cmake_minimum_required(VERSION 3.10)
project(RobotVision)

find_package(OpenCV REQUIRED)
find_package(Eigen3 REQUIRED)

add_subdirectory(src/detector)
add_subdirectory(src/tracker)

add_executable(run src/main.cpp)
target_link_libraries(run detector tracker ${OpenCV_LIBS})
```

这样模块化之后，改一个模块不用重新配置整个项目。

## 下一步

学完基础，你可能会问：既然手动 cmake/make 就能跑，为什么还有人用更高级的工具？

因为项目大了，依赖多了，手动管理就成噩梦了。Conan/vcpkg 管理依赖，CMake 的 install/export 机制管理安装，这些才是真正让你项目可以给别人跑起来的东西。

但先把基础打牢。那些都是后面自然而然的事。
